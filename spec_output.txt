3. Package 1 — cds-validate 
3.1  What To Build 
A middleware function that a CAP developer wraps around their request handler. Instead of writing 
50 lines of if-checks, they pass a Zod schema and this library does all the validation automatically. 
 
The exported API surface (what the developer uses): 
• 
validate(schema) — main middleware, validates req.data 
• 
validateQuery(schema) — validates query parameters 
• 
validateParams(schema) — validates action parameters 
• 
schemaFromEntity(entityName) — auto-builds a Zod schema from a CDS entity 
 
3.2  File-by-File Coding Plan 
File: src/types.ts — Write This First 
Define all TypeScript interfaces here before writing any logic. This file has no dependencies so it is 
the safest starting point. 
• 
CAPRequest interface — shape of the SAP req object (data, query, params, error function) 
• 
ValidationOptions interface — optional config like { strict: boolean, normalize: boolean } 
• 
CAPError interface — shape of the OData error object 
 
File: src/error-mapper.ts — Write Second 
Takes a Zod validation result and converts it into a CAP-compatible error object. 
• 
Input: Zod SafeParseError object 
• 
Output: CAP OData error with code, message, target, and details array 
• 
Target field: join the Zod error path array with forward slashes — this is already valid OData 
format 
• 
Details array: map every Zod error into { message, target } pairs 
• 
Use error code 422 (Unprocessable Entity) for validation failures 
 
File: src/validator.ts — The Core 
The main logic. This function returns another function (it is a factory/higher-order function). 
• 
Call schema.safeParse(req.data)

If it fails: call error-mapper, then call req.error() with the result — stop processing 
• 
If it passes: optionally overwrite req.data with the parsed result (this gives you type coercion

•

•

File: src/schema-builder.ts — Bonus Feature 
Reads the CDS compiled model and returns a Zod schema for a named entity. 
• 
Load the CDS model using cds.load() from @sap/cds 
• 
Find the entity by name in the loaded model 
• 
Loop through its elements (fields) and convert each type to Zod using the type map 
• 
Mark required vs optional based on CDS @mandatory annotation 
• 
Return the assembled z.object() schema 
 
File: src/index.ts — Last 
Just re-exports everything from the other files. Keep this file to 5–10 lines total.

4. Package 2 — cds-agents 
4.1  What To Build 
A library that reads your SAP CAP service definition, generates LangGraph tools for every entity 
and action automatically, and exposes them through a simple CAPAgent class that accepts plain 
English instructions. 
 
The exported API surface: 
• 
CAPAgent class — the main entry point, invoke() and stream() methods 
• 
CAPToolkit class — just the raw tools, no agent wrapper, for advanced users 
 
4.2  Phase-by-Phase Coding Plan 
Phase 1 — src/type-mapper.ts 
This is a pure lookup table. No HTTP calls, no CDS loading. Just a function that takes a CDS type 
string and returns the correct Zod type. 
 
CDS Type 
Zod Type 
Notes 
cds.String 
z.string() 
 
cds.UUID 
z.string().uuid() 
 
cds.Integer 
z.number().int() 
 
cds.Decimal 
z.number() 
Use z.coerce.number() for safety 
cds.Boolean 
z.boolean() 
 
cds.Date 
z.string() 
Format: YYYY-MM-DD 
cds.DateTime 
z.string().datetime() 
 
cds.LargeString 
z.string() 
No max length 
Unknown type 
z.any() 
Fallback — never crash 
 
Also handle associations (linked entities) — map them to z.string().uuid().optional() since they arrive 
as foreign keys. 
 
Phase 2 — src/odata-builder.ts

Builds OData v4 URL strings from plain inputs. This is a pure utility — no HTTP calls. 
• 
buildReadUrl(baseUrl, service, entity, options) — builds GET URL with $filter, $select, $top, 
$skip, $orderby, $expand 
• 
buildCreateUrl(baseUrl, service, entity) — builds POST URL 
• 
buildUpdateUrl(baseUrl, service, entity, id) — builds PATCH URL with key predicate 
• 
buildDeleteUrl(baseUrl, service, entity, id) — builds DELETE URL 
• 
buildActionUrl(baseUrl, service, actionName) — builds POST URL for actions 
• 
buildFilterString(conditions) — converts JS object to OData $filter syntax 
The hardest part is buildFilterString. Support at minimum: eq, lt, gt, le, ge, ne operators. Accept an 
object like { gpa: { lt: 2.0 }, status: 'active' } and output gpa lt 2.0 and status eq 'active'. 
 
Phase 3 — src/http-client.ts 
Makes the actual HTTP calls to the CAP service. Wraps Node.js fetch with CAP-specific handling. 
• 
Constructor accepts: baseUrl, auth config (basic or bearer token) 
• 
get(url) — GET request, returns parsed JSON or throws 
• 
post(url, body) — POST request 
• 
patch(url, body) — PATCH request 
• 
delete(url) — DELETE request 
• 
All methods add Authorization header automatically based on auth config 
• 
All methods check response.ok — if false, read the error body and throw a structured error 
• 
Parse OData error format: error.message and error.details from the response body 
 
Phase 4 — src/tool-generator.ts — The Most Important File 
This file reads the CDS model and creates all the LangGraph tool definitions. This is the heart of the 
entire library. 
 
Step A — Load the CDS model: 
• 
Call cds.load() with the path to the srv/ folder 
• 
Find the target service by name in csn.services 
• 
If service not found, throw a clear error: 'Service StudentService not found in CDS model' 
 
Step B — Generate CRUD tools for each entity: 
• 
Loop over Object.entries(service.entities) 
• 
For each entity, generate 4 tools using the LangGraph tool() function 
• 
READ tool: input schema has optional filter, select, top (number), skip (number), orderBy 
fields

• 
CREATE tool: input schema mirrors the entity fields from type-mapper 
• 
UPDATE tool: input schema has required id field + all entity fields as optional (partial 
update) 
• 
DELETE tool: input schema has required id field only 
• 
Each tool description must be specific, e.g.: 'Retrieves a list of Students. Use filter to narrow 
results by field values.' 
 
Step C — Generate tools for custom actions: 
• 
Loop over Object.entries(service.actions || {}) 
• 
Read the action's params from the CDS definition 
• 
Build input schema from params using type-mapper 
• 
Generate a single tool that POSTs to the action URL 
 
Step D — Apply entity filtering: 
• 
Accept an include array and exclude array in the config 
• 
If include is set, only generate tools for those entities 
• 
If exclude is set, skip those entities 
• 
Always skip entities whose name starts with underscore (internal CAP entities) 
 
Phase 5 — src/providers/anthropic.ts and openai.ts 
Thin wrappers that create the LangGraph-compatible LLM object from config. 
• 
anthropic.ts: takes model name and API key → returns ChatAnthropic instance from 
@langchain/anthropic 
• 
openai.ts: takes model name and API key → returns ChatOpenAI instance from 
@langchain/openai 
• 
Both export a createModel(config) function with identical signatures 
• 
Default model names: claude-sonnet-4-5 for Anthropic, gpt-4o for OpenAI 
 
Phase 6 — src/toolkit.ts 
The CAPToolkit class — for advanced users who want the raw tools but will build their own 
LangGraph graph. 
• 
Constructor: same config as CAPAgent 
• 
async init() — loads CDS model and generates tools, must be called before getTools() 
• 
getTools() — returns the raw array of LangGraph tool objects 
• 
This is also what CAPAgent uses internally

Phase 7 — src/agent.ts — The Final Wrapper 
The CAPAgent class that developers actually use. This should be the simplest file to write because 
all the hard work is done in the previous phases. 
• 
Constructor accepts: service, baseUrl, model, provider, tools, exclude, auth, maxSteps 
• 
async invoke(message) — calls the LangGraph ReAct agent with the message, returns final 
string response 
• 
async stream(message) — same but returns an async iterator for streaming tokens 
• 
Private method: createAgent() — loads toolkit, picks provider, creates ReAct agent from 
createReactAgent() in LangGraph 
• 
Lazy initialization — do not load CDS or connect until first invoke() call 
• 
maxSteps default: 10 — prevents infinite loops

6. Deployment Plan 
6.1  GitHub Repository Setup 
1. Create two public GitHub repositories: 
◦ 
github.com/YOUR_USERNAME/cds-validate 
◦ 
github.com/YOUR_USERNAME/cds-agents 
2. Add these files to each repo root: 
◦ 
README.md — usage docs with copy-paste examples 
◦ 
LICENSE — use MIT license 
◦ 
.gitignore — ignore node_modules, dist, .env 
◦ 
CHANGELOG.md — track versions 
3. Set up branch protection: require PR review before merging to main 
 
6.2  GitHub Actions CI/CD Pipeline 
Create .github/workflows/ci.yml in each package. This runs automatically on every push. 
 
The pipeline does 4 things in order: 
4. Install dependencies using pnpm 
5. Run TypeScript compiler to catch type errors 
6. Run all tests using Jest 
7. If on main branch AND a version tag was pushed: publish to npm automatically 
 
To trigger a release, you just run: git tag v1.0.0 && git push --tags
