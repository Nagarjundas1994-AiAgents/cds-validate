# cds-validate

![npm](https://img.shields.io/npm/v/cds-validate)
![license](https://img.shields.io/npm/l/cds-validate)

**Zod validation middleware for SAP CAP** — auto-validates request data with type coercion and OData-compliant error responses.

Stop writing 50 lines of `if`-checks in every CAP handler. Pass a Zod schema and let `cds-validate` handle the rest.

## Features

- 🛡️ **Schema validation** — validate `req.data`, `req.query`, and `req.params` with Zod
- 🔄 **Auto-partial on UPDATE** — `schema.partial()` applied automatically for PATCH/UPDATE events
- 🎯 **Type coercion** — automatically converts `"3"` → `3`, trims whitespace, etc.
- 📋 **OData errors** — returns proper OData error responses (422) with field-level details
- 🏗️ **Auto-schema generation** — build Zod schemas automatically from CDS entity definitions
- 📦 **Zero config** — works out of the box with any CAP project

## Installation

```bash
npm install cds-validate zod
```

> **Peer dependencies:** This package requires `@sap/cds` (>=7.0.0) and `zod` (>=3.20.0) to be installed in your project.

## Quick Start

```js
const { validate } = require('cds-validate');
const { z } = require('zod');

// Define your schema
const StudentSchema = z.object({
  name: z.string().min(1),
  email: z.string().trim().email(),
  gpa: z.coerce.number().min(0).max(4),
});

// Use in your CAP service
module.exports = (srv) => {
  // Validate before CREATE — all fields required
  srv.before('CREATE', 'Students', validate(StudentSchema));

  // Validate before UPDATE — only provided fields are checked
  srv.before('UPDATE', 'Students', validate(StudentSchema));
};
```

That's it. Invalid requests get a proper OData 422 error:

```json
{
  "error": {
    "code": 422,
    "message": "Validation failed with 2 errors",
    "target": "email",
    "details": [
      { "message": "Invalid email", "target": "email" },
      { "message": "Number must be less than or equal to 4", "target": "gpa" }
    ]
  }
}
```

## API Reference

### `validate(schema, options?)`

Creates a middleware that validates `req.data` against a Zod schema.

| Parameter | Type | Description |
|---|---|---|
| `schema` | `ZodSchema` | The Zod schema to validate against |
| `options.strict` | `boolean` | Reject unknown keys (default: `false`) |
| `options.normalize` | `boolean` | Overwrite `req.data` with parsed output (default: `true`) |

**Key behavior:**
- **CREATE** events: all schema fields are validated as defined.
- **UPDATE/PATCH** events: `schema.partial()` is applied automatically — only provided fields are validated.
- When `normalize: true`: `req.data` is replaced with the Zod-parsed result, giving you free type coercion and transforms.

```js
// With options
srv.before('CREATE', 'Students', validate(StudentSchema, {
  strict: true,      // reject unknown fields
  normalize: true,   // overwrite req.data with parsed output
}));
```

### `validateQuery(schema, options?)`

Validates `req.query` — use for OData query parameters.

```js
const QuerySchema = z.object({
  $top: z.coerce.number().max(100).optional(),
  $skip: z.coerce.number().min(0).optional(),
});

srv.before('READ', 'Students', validateQuery(QuerySchema));
```

### `validateParams(schema, options?)`

Validates `req.params` — use for bound action/function parameters.

```js
const EnrollParams = z.object({
  studentId: z.string().uuid(),
  courseId: z.string().uuid(),
});

srv.before('enrollStudent', validateParams(EnrollParams));
```

### `schemaFromEntity(entityName, options?)`

Auto-builds a Zod schema from a CDS entity definition. This is the **killer feature** — no need to manually write Zod schemas that mirror your CDS model.

| Parameter | Type | Description |
|---|---|---|
| `entityName` | `string` | Entity name (short or fully-qualified) |
| `options.model` | `CDSModel` | Pre-loaded CDS model (skips `cds.load()`) |
| `options.cdsFile` | `string` | Path to CDS files (default: `'./'`) |

```js
const { schemaFromEntity, validate } = require('cds-validate');

module.exports = async (srv) => {
  // Auto-generate schema from CDS model
  const StudentSchema = await schemaFromEntity('Students');

  srv.before('CREATE', 'Students', validate(StudentSchema));
  srv.before('UPDATE', 'Students', validate(StudentSchema));
};
```

**How it maps CDS types:**

| CDS Type | Zod Type |
|---|---|
| `cds.String` | `z.string()` |
| `cds.UUID` | `z.string().uuid()` |
| `cds.Integer` | `z.number().int()` |
| `cds.Decimal` | `z.coerce.number()` |
| `cds.Boolean` | `z.boolean()` |
| `cds.Date` | `z.string()` |
| `cds.DateTime` | `z.string().datetime()` |
| `cds.LargeString` | `z.string()` |
| Association | `z.string().uuid()` |
| Unknown | `z.any()` |

**Field rules:**
- `@mandatory` fields → **required**
- Key fields → **optional** (auto-generated by CAP)
- Virtual fields → **skipped**
- `_`-prefixed fields → **skipped** (internal CAP fields)

### `mapZodError(zodError)`

Low-level utility: converts a `ZodError` into a CAP OData error object. Used internally by `validate()`, but exported for custom use cases.

```js
const { mapZodError } = require('cds-validate');

const result = schema.safeParse(data);
if (!result.success) {
  const capError = mapZodError(result.error);
  req.error(capError); // { code: 422, message: '...', details: [...] }
}
```
